# CLAUDE.md — تحدي توقعات كأس العالم 2026

هذا الملف يمنحك السياق الكامل للمشروع. اقرأه قبل أي تعديل، واقرأ `docs/تحليل-النظام.md` لفهم المنطق التفصيلي للنظام.

## ما هو المشروع

منصة عربية (RTL) لتوقع نتائج مباريات كأس العالم 2026: كل عضو يتوقع النتيجة قبل المباراة، والتوقعات تُقفل قبل الانطلاق بـ 5 ثوانٍ، وينافس في تحدٍّ عام يضم الجميع + تحديات خاصة تُنشأ بكود من 6 خانات، مع لوحة أدمن.

## المعمارية

- **الواجهة**: React 18 + Vite، بدون مكتبة UI — التنسيق inline styles بهوية ثابتة (كحلي ليلي + ذهبي، خط Cairo). الألوان في `src/theme.js`.
- **الخلفية**: Supabase. **لا وصول مباشر للجداول من المتصفح** — RLS مفعّل بلا سياسات، وكل العمليات عبر دوال RPC بـ `security definer` في `supabase/schema.sql`.
- **الهوية**: اسم مستخدم + PIN (4–6 أرقام) بدون إيميل/جوال — قرار منتج نهائي، لا تقترح إضافة إيميل. الجلسة توكن UUID في localStorage.
- **بيانات الجدول**: `src/data/tournament.js` (الأسماء، المجموعات، المواعيد بتوقيت السعودية). بذرة قاعدة البيانات تُولَّد منها: `npm run seed` → `supabase/seed.sql`.
- معرّف المباراة موحّد بين الواجهة وقاعدة البيانات: `{iso}_{teamA}_{teamB}` مثل `2026-06-11_MX_ZA`.

## قواعد غير قابلة للكسر

1. **القفل الزمني يُفرض في الخادم فقط** (`submit_prediction` تتحقق من `now() >= kickoff_at - 5s`). العدّاد في الواجهة عرضي فقط. لا تنقل أي منطق قفل إلى العميل.
2. **النقاط ليست بيانات مخزّنة** — تُحسب دائمًا بدالة `match_points` من (التوقعات + النتائج). أي تصحيح نتيجة يحدّث كل اللوحات تلقائيًا. إن أضفت كاش للأداء لاحقًا فيجب أن يكون قابلًا لإعادة البناء بالكامل.
3. **توقع واحد لكل (عضو، مباراة)** يُحتسب في كل تحدياته — لا توقعات مختلفة لكل تحدٍّ.
4. **توقعات الأعضاء لا تنكشف قبل القفل** (`match_predictions` ترفض قبل `lock_at`).
5. عند تعديل أي شيء في `schema.sql`: الدوال كلها `create or replace` فيمكن إعادة لصق الملف كاملًا في SQL Editor بأمان. لا تكسر توقيعات الدوال المستخدمة في `src/lib/api.js` دون تحديثها معًا.
6. كل النصوص للمستخدم عربية، والتخطيط RTL.

## نظام النقاط (المعتمد)

دقيقة = 5 · اتجاه + فارق صحيح = 3 · اتجاه فقط = 1 · مضاعف المرحلة: qf ×2، sf/f ×3 · +2 للمتأهل الصحيح في الإقصائيات. كسر التعادل: النقاط ← الدقيقة ← الاتجاه. المنضم متأخرًا تُحسب له المباريات بعد `joined_at` فقط.

## بنية الملفات

```
supabase/schema.sql        السكيما + كل دوال RPC (المصدر الوحيد لمنطق الخادم)
supabase/seed.sql          مولَّد — لا تعدّله يدويًا، عدّل tournament.js ثم npm run seed
scripts/generate-seed.mjs  مولّد البذرة
src/lib/api.js             كل استدعاءات RPC + ترجمة الأخطاء للعربية + الجلسة
src/data/tournament.js     بيانات البطولة + ALL_MATCHES (مسطّحة بمعرّفات وأوقات ISO)
src/App.jsx                بوابة الدخول + التبويبات
src/components/            AuthScreen · ScheduleScreen (التوقع + العدّاد) ·
                           StandingsScreen · ChallengesScreen (لوحات الصدارة) · AdminScreen
src/legacy/                الملف الأصلي قبل التحويل — فيه كود بطاقات المشاركة Canvas
docs/تحليل-النظام.md       التحليل الكامل: الأدوار، الحالات الحرجة، خارطة الطريق
```

## حالة الإنجاز وخارطة الطريق

**منجز (المرحلتان 1 و 2 تقريبًا):** التسجيل والدخول، التوقع مع القفل والعدّاد، التحدي العام، إنشاء/انضمام التحديات الخاصة، لوحات الصدارة مع عدالة الانضمام المتأخر، كشف توقعات الأعضاء بعد القفل، اعتماد وتصحيح النتائج، إحصاءات المباراة وكشف توقعات آخر دقيقة، إعادة تعيين PIN، الحظر، سجل التدقيق (في القاعدة).

**التالي بالأولوية:**
1. إدارة التحدي الخاص لمنشئه: طرد عضو، تجديد الكود، قفل الانضمام، حذف التحدي (تحتاج دوال RPC جديدة: `kick_member`, `regenerate_code`, `lock_join`, `delete_challenge` — صلاحية المالك تُتحقق في الدالة).
2. بطاقات مشاركة المركز (واتساب/سناب) — انقل كود Canvas من `src/legacy/world-cup-schedule.original.jsx` (دالة `shareImageURL`) وكيّفه لبطاقة "مركزي في التحدي".
3. ترتيب الجولة/اليوم ("نجم اليوم") — دالة leaderboard بنطاق تاريخ.
4. توزيع التوقعات رسمًا بعد القفل (البيانات جاهزة من `admin_match_stats`؛ اعمل نسخة للأعضاء).
5. الأدوار الإقصائية: إضافة مبارياتها لقاعدة البيانات عند اكتمال المجموعات + حقل توقع المتأهل في الواجهة (الخادم جاهز: عمود `qualified` ومضاعفات المرحلة).
6. Realtime (اختياري): اشتراك Supabase Realtime لتحديث اللوحات لحظيًا بدل التحديث اليدوي.
7. عناصر الحماس: الجوكر، سلاسل التوقعات الدقيقة (راجع القسم 6 في التحليل).

## أوامر

```bash
npm install        # أول مرة
npm run dev        # تشغيل محلي (يتطلب .env من .env.example)
npm run build      # للتحقق قبل أي commit
npm run seed       # إعادة توليد seed.sql بعد تعديل الجدول
```

## ملاحظات تشغيلية

- أول أدمن يُعيَّن يدويًا في Supabase SQL Editor:
  `update profiles set is_admin = true where username = 'الاسم';`
- التحدي العام معرّفه ثابت `00000000-0000-0000-0000-000000000001` (تنشئه البذرة، وتعتمد عليه `register_user`). نفّذ seed.sql قبل أول تسجيل.
- الأوقات في seed.sql بصيغة `+03:00` (توقيت السعودية) وتُخزَّن timestamptz — المقارنات في الخادم صحيحة بغض النظر عن منطقة العضو الزمنية.
